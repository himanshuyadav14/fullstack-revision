Round 1 — React Router Theory (5 Questions)
R1 Q1 What is React Router? Why do we need it? ▾
Score: 9/10
Complete Answer

React Router is a client-side routing library. Maps URLs to components. Enables navigation without full page reloads using the History API (window.history.pushState). 

// Traditional routing — full page reload on every URL change
URL changes → GET request to server → server returns new HTML → full reload

// React Router — no server request
URL changes → JS intercepts → renders new component → instant, state preserved

// Three router types
BrowserRouter  // History API — /about (99% of apps)
HashRouter     // URL hash — /#/about (static hosting)
MemoryRouter   // In memory — React Native / tests
R1 Q2 BrowserRouter vs HashRouter — refresh behavior? ▾
Score: 9/10 
// BrowserRouter — clean URLs, needs server config
URL: myapp.com/about
Refresh → browser sends GET /about to server
Server has no /about → 404 ❌
Fix: configure server → return index.html for ALL routes
// Nginx: try_files $uri /index.html

// HashRouter — hash URLs, no server config needed
URL: myapp.com/#/about
Refresh → browser sends GET / (ignores everything after #)
Server returns index.html ✅ → React Router reads #/about

// When to use each:
BrowserRouter  // production apps with server config (default)
HashRouter     // GitHub Pages, S3, static hosting without server config

// ProtectedRoute.jsx — Outlet pattern
const ProtectedRoute = () => {
  const { isAuthenticated } = useContext(AuthContext)
  const location = useLocation()

  return isAuthenticated
    ? <Outlet />
    : <Navigate to="/login" replace state={{ from: location.pathname }} />

  // state.from — login can redirect back after success
}

Three Redux Principles — Always Say These
// 1. Single source of truth
// Entire app state in ONE store

// 2. State is read-only
// Only way to change state is dispatch(action)

// 3. Changes by pure reducers
reducer(currentState, action) = newState  // no side effects

// Redux data flow — always unidirectional
UI Event → dispatch(action) → reducer → new state → UI updates

// Context vs Redux
Context    // convenience (no prop drilling), all consumers re-render
Redux      // granular subscriptions, middleware, devtools, time-travel debug

R2 Q5 createAsyncThunk — async API call with pending/fulfilled/rejected states ▾
Score: 0/10 — Not attempted
Complete Pattern — Memorize This
// 1. Define async thunk
export const fetchUsers = createAsyncThunk(
  'users/fetchUsers',
  async () => {
    const res = await fetch('https://jsonplaceholder.typicode.com/users')
    if (!res.ok) throw new Error(`HTTP error: ${res.status}`)
    return res.json()  // becomes action.payload on success
  }
)

// 2. Handle 3 states in slice
const usersSlice = createSlice({
  name: 'users',
  initialState: { data: [], loading: false, error: null },
  reducers: {},
  extraReducers: (builder) => {
    builder
      .addCase(fetchUsers.pending, (state) => {
        state.loading = true
        state.error = null
      })
      .addCase(fetchUsers.fulfilled, (state, action) => {
        state.loading = false
        state.data = action.payload
      })
      .addCase(fetchUsers.rejected, (state, action) => {
        state.loading = false
        state.error = action.error.message
      })
  }
})

// 3. Component
useEffect(() => { dispatch(fetchUsers()) }, [dispatch])
const { data, loading, error } = useSelector(state => state.users)
Three states generated automatically
fetchUsers.pending → API call started
fetchUsers.fulfilled → success → action.payload = returned data
fetchUsers.rejected → failed → action.error.message = error

One line distinction

R4 Q2 Redux slice with push/splice/find mutations — is this correct? ▾
Score: 10/10
The Trap — Most People Get This Wrong
Code IS correct — RTK uses Immer, mutation syntax is intentional and safe
DON'T say "can't mutate state" — that's outdated plain Redux thinking 
// ✅ Valid in RTK — Immer handles immutability
state.users.push(action.payload)
state.users.splice(index, 1)
user.name = action.payload.name

// Real bugs — edge cases:
// Bug 1: splice(-1) removes LAST element when findIndex returns -1
const index = state.users.findIndex(u => u.id === action.payload)
if (index !== -1) state.users.splice(index, 1)  // ← guard required

// Bug 2: user.name crashes if user not found
const user = state.users.find(u => u.id === action.payload.id)
if (user) user.name = action.payload.name  // ← guard required

// Alternative deleteUser — cleaner, no index bug
state.users = state.users.filter(u => u.id !== action.payload)

// ❌ Plain Redux — counter (lots of boilerplate)
const INCREMENT = 'counter/increment'
const increment = () => ({ type: INCREMENT })

    default: return state
