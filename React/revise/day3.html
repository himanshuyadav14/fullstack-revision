Round 1 — Theory Fundamentals (5 Questions)
R1 Q1 What problem do useMemo and useCallback actually solve? Why do they exist? ▾
Score: 7/10
Root Cause — Referential Equality
// Primitives compared by VALUE
1 === 1  // true

// Objects/functions compared by REFERENCE
{} === {}            // false — new object each render
() => {} === () => {} // false — new function each render

// Problem 1 — expensive recalculation every render
const result = heavyCalc(data)  // runs on every render ❌
const result = useMemo(() => heavyCalc(data), [data])  // cached ✅

// Problem 2 — unstable function reference breaks React.memo
const fn = () => {}  // new ref every render ❌
const fn = useCallback(() => {}, [])  // stable ref ✅
When NOT to use
Both have comparison overhead every render — don't add by default
useMemo only for expensive calculations. useCallback only when child uses React.memo
useCallback(fn, deps) is literally useMemo(() => fn, deps) under the hood
R1 Q2 React.memo vs useMemo — difference? Custom comparator? ▾
Score: 6/10
Distinction

React.memo — HOC wrapping a component. Skips re-render if props unchanged (shallow compare).

useMemo — Hook inside component. Caches computed value until deps change. 

// React.memo wraps component
const Child = React.memo(({ user }) => <div>{user.name}</div>)

// React.memo with custom comparator
const Child = React.memo(Component, (prev, next) => {
  return prev.user.id === next.user.id
  // true = skip re-render, false = re-render
})

// useMemo inside component
const sorted = useMemo(() => [...users].sort(...), [users])
Shallow Compare Trap
Object literal prop: <Child user={{ name: 'Alice' }} /> — new ref every render — React.memo useless
Fix: const user = useMemo(() => ({ name: 'Alice' }), [])

R1 Q5 useContext vs Redux/Zustand — when to use each? Context limitations? ▾
Score: 6.5/10
When to Use Each
// Context — theme, current user, locale, feature flags
// Static or rarely changing global values

// Zustand/Redux — shopping cart, real-time data,
// frequent updates, middleware needed, devtools needed

// Zustand granular subscription — Context cannot do this
const count = useStore(state => state.count)
// only re-renders when count changes, not other state
Context Limitations
No granular subscriptions — can't subscribe to just one field
No middleware — can't log/intercept actions
No devtools — can't time-travel debug
Context solves CONVENIENCE, not PERFORMANCE

R2 Q3 What is a custom hook? Rules? Share logic not state? ▾

// Rules: start with "use", top level only, only inside React fn or custom hook
function useCounter() {
  const [count, setCount] = useState(0)
  const increment = useCallback(() => setCount(c => c + 1), [])

}
CRITICAL — Hooks share LOGIC not STATE
ComponentA and ComponentB each calling useCounter() get INDEPENDENT state
Changing count in A does NOT affect B
For shared state → use Context or state management library

const ThemeContext = createContext()

function ThemeProvider({ children }) {
  const [theme, setTheme] = useState('light')
  const toggleTheme = useCallback(() =>
    setTheme(prev => prev === 'light' ? 'dark' : 'light')
  , [])
  const value = useMemo(() => ({ theme, toggleTheme }), [theme, toggleTheme])
  return <ThemeContext.Provider value={value}>{children}</ThemeContext.Provider>
}

function useTheme() {
  const ctx = useContext(ThemeContext)
  if (!ctx) throw new Error('useTheme must be used within ThemeProvider')
  return ctx
}

// Usage
function Header() {
  const { theme, toggleTheme } = useTheme()
  return (
    <div style={{ background: theme === 'light' ? '#fff' : '#333' }}>
      <p>Theme: {theme}</p>
      <button onClick={toggleTheme}>Toggle</button>
    </div>
  )
}
R3 Q2 useArray — push, remove, update, clear, reset ▾
Score: 9/10 
export function useArray(initialValue) {
  const [arr, setArr] = useState([...initialValue])
  const initialRef = useRef([...initialValue])

  const push = item => setArr(prev => [...prev, item])
  const remove = index => setArr(prev => prev.filter((_, i) => i !== index))
  const update = (index, item) =>
    setArr(prev => prev.map((el, i) => i === index ? item : el))
  const clear = () => setArr([])
  const reset = () => setArr([...initialRef.current])

  return { arr, push, remove, update, clear, reset }
}

// Reducer OUTSIDE component
function cartReducer(state, action) {
  switch (action.type) {
    case 'ADD_ITEM': {
      const exists = state.find(i => i.id === action.payload.id)
      if (exists) return state.map(i =>
        i.id === action.payload.id ? { ...i, quantity: i.quantity + 1 } : i)
      return [...state, { ...action.payload, quantity: 1 }]
    }
    case 'REMOVE_ITEM':
      return state.filter(i => i.id !== action.payload)
    case 'UPDATE_QUANTITY':
      return state.map(i =>
        i.id === action.payload.id
          ? { ...i, quantity: Math.max(1, action.payload.quantity) }
          : i)
    case 'CLEAR_CART':
      return []
    default:
      return state
  }
}

R3 Q4 useMultiStep — next, back, goTo, isFirst, isLast + 3 step form ▾
Score: 9.5/10 
export function useMultiStep(steps) {
  const [currentStep, setCurrentStep] = useState(0)

  const next = () =>
    setCurrentStep(prev => prev < steps.length - 1 ? prev + 1 : prev)
  const back = () =>
    setCurrentStep(prev => prev > 0 ? prev - 1 : prev)
  const goTo = step => {
    if (step >= 0 && step <= steps.length - 1) setCurrentStep(step)
  }

  return {
    currentStep,
    step: steps[currentStep],
    next, back, goTo,
    isFirst: currentStep === 0,
    isLast: currentStep === steps.length - 1
  }
}

R4 Q3 UserList — 3 bugs: mutation + no useMemo + inline onClick ▾
Score: 9/10 
// Bug 1 — users.sort() MUTATES original array in place (silent, dangerous)
// Bug 2 — no useMemo — sorts on every render
// Bug 3 — inline onClick — new function per item per render

// ✅ Complete fix
const ListItem = React.memo(({ user, onSelect }) => (
  <li onClick={() => onSelect(user)}>{user.name}</li>
))

function UserList({ users, onSelectUser }) {
  const sortedUsers = useMemo(() =>
    [...users].sort((a, b) => a.name.localeCompare(b.name))  // spread first!
  , [users])

  const handleSelect = useCallback(user => {
    onSelectUser(user)
  }, [onSelectUser])

  return (
    <ul>
      {sortedUsers.map(user => (
        <ListItem key={user.id} user={user} onSelect={handleSelect} />
      ))}
    </ul>
  )
}

R4 Q4 React.lazy + Suspense + Error Boundary for heavy dashboard ▾
Score: 6/10
What was missed
Each lazy component must be in its OWN file — different files = different code chunks
Suspense does NOT catch load failures — need Error Boundary
Error Boundary MUST be a class component — no hook equivalent 
// Each in its OWN file
const Analytics = React.lazy(() => import('./Analytics'))
const Revenue = React.lazy(() => import('./Revenue'))

// Error Boundary — class component only
class ErrorBoundary extends React.Component {
  state = { hasError: false }
  static getDerivedStateFromError() { return { hasError: true } }
  render() {
    if (this.state.hasError) return <p>Failed. Refresh.</p>
    return this.props.children
  }
}

// Wrap each lazy component
<ErrorBoundary>
  <Suspense fallback={<p>Loading...</p>}>
    <Analytics />
  </Suspense>
</ErrorBoundary>
