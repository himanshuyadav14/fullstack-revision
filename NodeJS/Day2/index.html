Q01 What is Express.js? Why was it created? What does it give you over raw http module? 9.5/10 ▶
Key Answer

Express.js is a minimal, lightweight web framework built on top of Node.js. Created because raw http module becomes repetitive and hard to maintain in real apps. Express provides middleware architecture, clean routing, enhanced req/res, and code organization.

http vs Express comparison 
// Raw http — repetitive, manual
if (req.url === '/about' && req.method === 'GET') {
  res.end('About')
}

// Express — clean, automatic
app.get('/about', (req, res) => res.send('About'))
What Express adds
Middleware architecture — app.use(express.json())
Enhanced req — req.params, req.query, req.body
Enhanced res — res.send(), res.json(), res.status()
Code organization — Routers, Controllers, Middleware, Services
Missing point
req.params auto-parsed — in raw http you split URL manually: req.url.split('/')[2]. Express does it automatically with :id syntax.
Q02 What is the middleware chain? How does next() work? What if you forget next() or res.send()? 9.5/10 ▶
Middleware Chain Flow
// Request → MW1 → MW2 → MW3 → Response
app.use((req, res, next) => {
  console.log('MW1')
  next()  // pass to next middleware
})

app.use((req, res, next) => {
  console.log('MW2')
  // next() forgotten → REQUEST HANGS
})
Critical bugs to remember
Forget next() → request hangs forever, client keeps loading. No response sent.
Forget res.send() → request reaches route but client still waits. Same hang.
Call both next() AND res.send() → "Cannot set headers after they are sent" error.
Always return before res.send() 
// ✅ ALWAYS return to prevent next() running after
app.use((req, res, next) => {
  if (!req.headers.authorization) {
    return res.status(401).json({ error: 'Unauthorized' })
  }
  next()
})

// Request timeout in production
res.setTimeout(5000, () => res.status(408).json({ error: 'Timeout' }))

Q04 How does error handling work? Sync vs async errors. Write error handler middleware. 9.5/10 ▶
Sync vs Async error handling 
// Sync — Express catches automatically
app.get('/', (req, res) => {
  throw new Error('Sync error')  // auto caught
})

// Async — MUST use next(err) in Express 4
app.get('/', async (req, res, next) => {
  try {
    const data = await fetchDB()
    res.json(data)
  } catch (err) {
    next(err)  // manually pass to error handler
  }
})
asyncHandler pattern — production must-know 
// Wrap routes to avoid repeating try/catch
const asyncHandler = (fn) => (req, res, next) =>
  Promise.resolve(fn(req, res, next)).catch(next)

// Usage — no try/catch needed!
app.get('/users', asyncHandler(async (req, res) => {
  const users = await User.find()
  res.json(users)
}))
Complete error handler
const errorHandler = (err, req, res, next) => {
  let statusCode = err.statusCode || 500
  let message = err.message || 'Internal Server Error'

  if (err.name === 'ValidationError') {
    statusCode = 400; message = 'Validation Error'
  }
  if (err.name === 'JsonWebTokenError' || err.name === 'TokenExpiredError') {
    statusCode = 401; message = 'Unauthorized'
  }

  res.status(statusCode).json({
    success: false, message,
    stack: process.env.NODE_ENV === 'production' ? null : err.stack
  })
}

Q08 What is CORS? Simple request vs preflight request? How to configure in Express? 7.5/10 ▶
What is Origin?
// Origin = Protocol + Domain + Port
http://localhost:3000  ↔  http://localhost:5000  // DIFFERENT (port)
http://myapp.com     ↔  https://myapp.com     // DIFFERENT (protocol)
http://api.myapp.com ↔  http://myapp.com      // DIFFERENT (subdomain)
Preflight request — what you missed 
// Simple Request (GET, HEAD, POST with basic headers):
Browser → Server  // direct, no check needed

// Preflight Request (PUT, DELETE, PATCH or custom headers):
Browser → Server  // OPTIONS request first: "are you okay with this?"
         ↓
Server responds with allowed methods/headers
         ↓
Browser → Server  // actual request
// 2 round trips!
Full CORS config
app.use(cors({
  origin: ['http://localhost:3000', 'https://myapp.com'],
  methods: ['GET', 'POST', 'PUT', 'DELETE'],
  allowedHeaders: ['Content-Type', 'Authorization'],
  credentials: true  // allow cookies cross-origin
}))
Simple vs Preflight table
Simple Request Preflight
Methods GET, HEAD, POST PUT, DELETE, PATCH
Custom headers Basic only Authorization etc.
Browser Direct request OPTIONS check first
Round trips 1 2

Correct controller pattern

  { id: 1, name: 'John' },
  { id: 2, name: 'Jane' }
]

const getUserById = (req, res) => {
  const userId = Number(req.params.id)  // ← convert!
  const user = users.find(u => u.id === userId)
  if (!user) return res.status(404).json({ message: 'Not found' })
  res.json(user)

const deleteUser = (req, res) => {
  const userId = Number(req.params.id)  // ← convert!
  const user = users.find(u => u.id === userId)
  if (!user) return res.status(404).json({ message: 'Not found' })
  users = users.filter(u => u.id !== userId)  // ← let!

Q13 requestLogger middleware — log method, URL, status code, time taken 4/10 ▶
Critical bugs
Bug 1 — Logging status code in middleware runs BEFORE route. Status is always 200 at that point. Must log AFTER response sent.
Bug 2 — res.responseTime does not exist. Must calculate manually with Date.now().
Correct solution — res.on('finish') 
const requestLogger = (req, res, next) => {
  const start = Date.now()

  res.on('finish', () => {  // fires AFTER response sent
    const duration = Date.now() - start
    console.log(`${req.method} ${req.url} ${res.statusCode} ${duration}ms`)
    // POST /api/users 201 45ms
  })

  next()
}
// This is exactly how morgan works internally!
Key insight

res.on('finish') is an event that fires after response is fully sent to client. Only at this point do you know the real status code and real duration.

Q14 validateUser middleware — name, email, optional age validation 10/10 ▶
Perfect solution
const validateUser = (req, res, next) => {
  const { name, email, age } = req.body
  const errors = []

  // name
  if (!name) errors.push('Name is required')
  else if (typeof name !== 'string') errors.push('Name must be a string')
  else if (name.trim().length < 2) errors.push('Name min 2 chars')

  // email
  if (!email) errors.push('Email is required')
  else if (!/^[^\s@]+@[^\s@]+\.[^\s@]+$/.test(email)) errors.push('Invalid email')

  // age (optional)
  if (age !== undefined) {
    if (typeof age !== 'number') errors.push('Age must be a number')
    else if (age < 18 || age > 100) errors.push('Age must be 18-100')
  }

  if (errors.length > 0)
    return res.status(400).json({ success: false, errors })

  next()
}

Q16 JWT auth middleware — extract Bearer token, verify, attach req.user 10/10 ▶
Complete auth middleware
const jwt = require('jsonwebtoken')

const auth = (req, res, next) => {
  try {
    const authHeader = req.headers.authorization
    if (!authHeader)
      return res.status(401).json({ message: 'Header missing' })

    if (!authHeader.startsWith('Bearer '))
      return res.status(401).json({ message: 'Invalid format' })

    const token = authHeader.split(' ')[1]
    const decoded = jwt.verify(token, process.env.JWT_SECRET)
    req.user = decoded
    next()
  } catch (err) {
    return res.status(401).json({ message: 'Invalid or expired token' })
  }
}

// Usage
app.get('/profile', auth, (req, res) => res.json({ user: req.user }))

Q20 Handler A vs B — next() vs next(err) vs next('route') — async bug 10/10 ▶
Part A — Express checks middleware.length 
// Handler A — 3 params — treated as NORMAL middleware ❌
app.use((err, req, res) => { ... })
// Express checks: middleware.length === 4
// 3 params → not recognized as error handler

// Handler B — 4 params — recognized as error handler ✅
app.use((err, req, res, next) => { ... })
Part B — next() vs next(err) vs next('route') 
next()         // → go to next middleware normally
next(err)      // → skip everything, jump to error handler
next('route')  // → skip remaining handlers for THIS route,
               //   go to next matching route definition

// next('route') example:
app.get('/user',
  (req, res, next) => { next('route') },  // skipped
  (req, res) => { res.send('First') }     // skipped
)
app.get('/user', (req, res) => res.send('Second')) // runs!
Part C — async error not caught in Express 4 
// ❌ Bug — unhandled promise rejection in Express 4
app.get('/data', async (req, res, next) => {
  const data = await fetchFromDB()  // throws!
  // error middleware never called — request hangs
})

// ✅ Fix — wrap in try/catch + next(err)
app.get('/data', async (req, res, next) => {
  try {
    const data = await fetchFromDB()
    res.json(data)
  } catch (err) {
    next(err)  // passes to error middleware
  }
})
