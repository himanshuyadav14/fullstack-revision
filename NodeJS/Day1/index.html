Q01 What is Node.js? Why was it created? How is it different from running JavaScript in a browser? 3/10 ▾
✅ What was right
  • ✓Node.js is a JavaScript runtime environment — correct
⚠️ What was missing
  • !Why it was created — Ryan Dahl built it in 2009 to solve slow, blocking web servers (Apache etc. blocked one thread per request)
  • !V8 Engine — Node.js runs on Chrome's V8 engine which compiles JS to machine code
  • !Browser vs Node differences — DOM, window, document not available in Node; Node has fs, OS access
  • !"Anywhere in watch" — too vague. Correct: server, terminal, IoT devices
📖 Complete Answer
Browser JS Node.js
Has DOM access No DOM
window, document exist global object instead
Cannot access files Full file system access (fs)
Security sandbox OS-level access

One-liner: Node.js is a JavaScript runtime built on Chrome's V8 engine, created in 2009 by Ryan Dahl to build fast, non-blocking servers. Unlike browser JS, it has no DOM but has access to file system, network, and OS-level features.

Q02 What is the Event Loop? Why does it exist? How does Node.js handle thousands of requests without creating thousands of threads? 8.5/10 ▾
✅ What was right
  • ✓Single-threaded + Event Loop concept
  • ✓Call stack, callback queue, microtask queue priority
  • ✓libuv offloads I/O — most people miss this!
  • ✓Thread explosion problem explained
⚠️ What was missing
  • !process.nextTick() has HIGHER priority than Promise callbacks inside microtask queue
  • !Event Loop has 6 phases: Timers → Pending → Idle → Poll → Check (setImmediate) → Close
  • !setImmediate vs setTimeout(fn,0) — setImmediate always wins inside I/O callbacks
📖 Priority Order
// Execution priority (highest → lowest)
1. Synchronous code (call stack)
2. process.nextTick()        // highest async priority
3. Promise.then() callbacks   // microtask queue
4. setTimeout / setInterval   // timers phase
5. setImmediate               // check phase
6. I/O callbacks              // poll phase

Q05 fs module — readFile vs readFileSync — when to use which? 6.5/10 ▾
⚠️ Most important rule
  • !NEVER use readFileSync inside a route handler — it blocks the ENTIRE event loop for all users
  • !readFileSync is OK ONLY at server startup — reading config.json before app boots
📖 Correct Pattern
// ❌ NEVER in route handler
app.get('/data', (req, res) => {
  const data = fs.readFileSync('file.txt') // blocks everyone!
})

// ✅ Use async
app.get('/data', async (req, res) => {
  const data = await fs.promises.readFile('file.txt', 'utf-8')
  res.send(data)
})

// ✅ readFileSync OK at startup only
const config = JSON.parse(fs.readFileSync('config.json', 'utf-8'))
app.listen(config.port)

Q14 Read large file with streams — count total lines, handle errors 10/10 ▾
✅ Perfect Solution
const fs = require('fs')

const readLargeFile = (filePath) => {
  let totalLines = 0
  let leftOver = ''
  const stream = fs.createReadStream(filePath, { encoding: 'utf-8' })

  stream.on('data', (chunk) => {
    let data = leftOver + chunk
    let lines = data.split('\n')
    leftOver = lines.pop() // save incomplete last line
    totalLines += lines.length
  })

  stream.on('end', () => {
    if (leftOver) totalLines++ // handle no trailing newline
    console.log(`Total lines: ${totalLines}`)
  })

  stream.on('error', (err) => console.error(err.message))
}
📖 Key insight

The leftOver variable is the critical insight. Chunks don't always end at \n. The last incomplete line must carry over to the next chunk — otherwise line count is wrong.

Q15 Implement EventEmitter from scratch — on, emit, off, once 8.5/10 ▾
⚠️ Bugs in your solution
  • !emit() — used listener(args) instead of listener(...args) — args passed as array not spread
  • !once() wrapper — didn't pass ...args to listener()
📖 Correct Solution
class EventEmitter {
  constructor() { this.events = {} }

  on(event, listener) {
    if (!this.events[event]) this.events[event] = []
    this.events[event].push(listener)
  }

  emit(event, ...args) {
    if (!this.events[event]) return
    this.events[event].forEach(l => l(...args)) // ← spread!
  }

  off(event, toRemove) {
    if (!this.events[event]) return
    this.events[event] = this.events[event].filter(l => l !== toRemove)
  }

  once(event, listener) {
    const wrapper = (...args) => { // ← spread args!
      listener(...args)
      this.off(event, wrapper)
    }
    this.on(event, wrapper)
  }
}
Q16 Read JSON → add updatedAt to each object → write back to file 10/10 ▾
✅ Perfect Solution
const { readFile, writeFile } = require('fs').promises

async function updateJson(filePath) {
  try {
    const fileData = await readFile(filePath, 'utf-8')

    let users
    try {
      users = JSON.parse(fileData)
    } catch (err) {
      console.error('Invalid JSON:', err.message)
      return
    }

    if (!Array.isArray(users)) return console.error('Not an array')

    console.error('Error:', err.message)
  }
}

Q17 Implement promisify from scratch 10/10 ▾
✅ Perfect Solution
function promisify(fn) {
  return function (...args) {
    return new Promise((resolve, reject) => {
      fn(...args, (err, data) => {
        if (err) reject(err)
        else resolve(data)
      })
    })
  }
}
// This is exactly how util.promisify works internally!
Q18 Implement rate limiter middleware — 5 req/min per IP, return 429 9.5/10 ▾
✅ Your Solution was correct 
const requestStore = new Map()

function rateLimiter(req, res, next) {
  const ip = req.ip
  const now = Date.now()
  const WINDOW = 60 * 1000
  const MAX = 5

  if (!requestStore.has(ip)) {
    requestStore.set(ip, { count: 1, startTime: now })
    return next()
  }

  const data = requestStore.get(ip)

  if (now - data.startTime > WINDOW) {
    requestStore.set(ip, { count: 1, startTime: now })
    return next()
  }

  data.count++
  if (data.count > MAX) return res.status(429).json({ error: 'Too Many Requests' })
  next()
}

// Memory leak fix — cleanup expired IPs
setInterval(() => {
  const now = Date.now()
  for (const [ip, data] of requestStore.entries()) {
    if (now - data.startTime > 60000) requestStore.delete(ip)
  }
}, 60000)

Q21 forEach with async — what's wrong? Fix with Promise.all 10/10 ▾
⚠️ The bug
// ❌ forEach does NOT await async callbacks
files.forEach(async (file) => {
  const data = await fs.readFile(file)  // runs but nobody waits
  results.push(data)
})
return results  // returns [] immediately!
📖 Fix + Promise.allSettled
// ✅ Promise.all — all must succeed
const results = await Promise.all(
  files.map(f => fs.readFile(f, 'utf-8'))
)

// ✅ Promise.allSettled — continue even if some fail
const results = await Promise.allSettled(
  files.map(f => fs.readFile(f, 'utf-8'))
)
results.forEach(r => {
  if (r.status === 'fulfilled') console.log(r.value)
  else console.error(r.reason)
})
Q22 CPU intensive work blocking event loop — 3 ways to fix 0/10 ▾
⚠️ The problem

A for loop with 10 billion iterations runs on the call stack and NEVER yields. Event loop is completely blocked. /ping will be stuck until the loop finishes.

📖 Fix 1 — Worker Threads (best) 
const { Worker } = require('worker_threads')

app.get('/hash', (req, res) => {
  const worker = new Worker(`
    const { workerData, parentPort } = require('worker_threads')
    let result = 0
    for (let i = 0; i < workerData.limit; i++) result += i
    parentPort.postMessage(result)
  `, { eval: true, workerData: { limit: 10_000_000_000 } })

  worker.on('message', result => res.json({ result }))
})
📖 Fix 2 — setImmediate chunking 
function processChunk(i, result, res) {
  const end = Math.min(i + 1_000_000, 10_000_000_000)
  while (i < end) result += i++
  if (i < 10_000_000_000)
    setImmediate(() => processChunk(i, result, res)) // yield!
  else
    res.json({ result })
}
📖 Which to use?
Fix Use when
Worker Threads CPU intensive, need shared memory
Child Process Heavy isolated tasks
setImmediate chunks Simple chunking, no extra processes
